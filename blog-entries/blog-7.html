Web Services - Are You Speaking a Foriegn Language?

Now, I’m sure you’ve at least heard of the name Amazon Web Services (AWS), Microsoft Azure, or Google Cloud Platform (GCP). These are all major web services.

From a broad perspective, a web service is a software application providing protocols (HTTP/HTTPS) to communicate, share, and exchange data over a network. Most commonly web services are XML-data oriented. A positive of web services, is for the ability of applications to be written in different languages, but still have the ability to communicate and exchange information to other applications through servers and clients. A web service hides implementation details in which it can be used separately in hardware or software contexts through an interface.

The following features make up a web service:

  • Accessible over internet/intranet networks
  • Includes a standardized XML messaging system
  • Independent of a OS or programming language
  • Self-describing via standard XML language
  • Found through a simple location

Are There any Varying Calibers of Web Services?

There are quite a few out there. I'll name several below:

XML-RPC (Remote Procedure Call) - claimed as the most basic XML protocol to exchange data over a broad scale of devices on a network. XML-RPC uses HTTP to painlessly transfer data and communicate information from client to server.

UDDI (Universal Description, Discovery, and Integration) - is an XML standard for detailing, publishing, as well as discovering web services. Simply put, It’s an internet registry for businesses around the globe. This standard has the goal in mind to streamline digital transactions/commerce among company systems.

SOAP - is an XML protocol used to exchange data and documents over HTTP or SMTP (Simple Mail Transfer Protocol). SOAP allows for independent processes operating on differing systems to communicate through XML.

REST - provides communication and connectivity between devices and the internet for API-like tasks. RESTful services typically will use HTTP as the supporting protocol.

SOAP and REST are some of the most popular choices among developers. There’s been a lot of debate as to which one is the winner over all the competition, although, it seems to be the context for when it’s being used matters most of all. A REST web service is often great for when time is a factor, but SOAP seems to be best when building a service with many, non-CRUD methods. Oftentimes, companies will support both choices, including Amazon and eBay.

Could You Explain the Architecture of Web Services?

In order for the framework to function ideally there needs to be some architecture to support it. There are three distinct roles in the architecture of web services which all assist in finding, binding, and publishing the web service:

  1. Service Provider: creates the web service and makes it available to the client application to use it for its own specific purposes.
  2. Service Requestor: known as the client application that is needed to contact the web service. Popular service requestors include .Net, Java which search for a particular kind of functionality through a web service.
  3. Service Broker: the application that gives access to the UDDI (Universal Description, Discovery, and Integration), providing a repository on which various XML notation files for describing the service (including WSDL) may be hosted.

Wait a Sec, I’ve Heard the Term “API” Thrown Around a Bit. Is an API and a Web Service the Same Thing?

Kinda, but not exactly. An API or application programming interface offers a service to other pieces of software, forming a connection between computers/programs. Most web services provide an API, which, with its set of commands and functions, is used to retrieve data. An example of this includes Twitter, which delivers an API authorizing a developer to access tweets from a server and then collects data in JSON format.
